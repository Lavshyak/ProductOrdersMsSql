.NET 10, ASP.NET, Entity Framework (LINQ, raw SQL), MS SQL Server 2022, Docker compose, xUnit, testcontainers. Многопоточное, асинхронное, конкурентное.

Назначение сервиса - минимизация запросов остатка товара у поставщиков, минимизация попыток продажи сверх запасов.\
Хранит остатки товаров и резервирования. \
Предназначен для приватного использования бэкендом.

Разработан с упором на среднюю-высокую нагрузку. (использую MS SQL для этого просто потому что пет проект).\
Минимальная критичная функциональность протестирована.\
Можно шардировать/горизонтально скалировать эту связку (сервер+СУБД) по ProductId, разработав дополнительный сервис-оркестратор.

- POST /stock/set - установить имеющееся количество товара у поставщика.
- POST /reservations - попытка добавить резервирование и уменьшить остаток товара.
- POST /reservations/cancel - отменить резервирование, вернуть товар в остаток.
- POST /reservations/writeoff - удалить резервирование, не увеличивая остаток (списание товара).
- GET  /stock/{productId} - получить количество незарезервированного товара.

WEB API принимает и возвращает только GUID v7. Более подробная документация WEB API в файле `openapi.json`.

В таблице OutboxMessages сервис "уведомляет" о побочно отмененных резервированиях 
(при уменьшении имеющегося кол-ва товара, при отмене истекших резервирований)\
***Столбцы:***\
EventType: `reservation.cancelled.expired` | `reservation.cancelled.notenoughstock`,\
OccuredAtUtc,\
PayloadJson:`{"ReservationId":"BD6F721C-3A89-DE9F-7AF6-019DFE3E4D44"}` - MS SQL Server Friendly GUIDv7

Ограничения:
- Во всей БД, в том числе в таблице OutboxMessages, хранятся MS SQL Server Friendly GUIDv7 в качестве идентификаторов. Перед использованием во внешних сервисах нужно сконвертировать их обратно в обычный GUID v7. При этом WEB API сервиса принимает и возвращает только обычные GUID v7, сервис занимается конвертацией сам.
- При понижении TotalQuantity товара может отмениться небольшое количество резервирований, для которых хватает остатка товара.
- Сервис не предоставляет возможности отменить отмену резервирования (откатить).
- Сервис не предоставляет возможности изменить количество товара, добавить или удалить товар из уже существующего резервирования.
- Сервис не уведомляет об изменении незарезервированного количества товара, фактического количества товара, добавлении или удалении резервирования по запросу.
- Сервис не хранит историю каких-либо изменений, кроме `reservation.cancelled.expired` и `reservation.cancelled.notenoughstock`.
- При попытке добавления резервирования, если резервирование с таким Id существует, то сервис всегда будет возвращать ошибку.
- Идемпотентность не гарантируется, но в некоторых случаях имеется.

Внутренние технические моменты:
- В качестве идентификаторов в СУБД используется GUID v7, байты которого переставлены в порядке сортировки, оптимизированном для MS SQL Server. Swapping туда-обратно в Shared/GuidExtensions.cs.
- ReservationItems имеет FK на Reservations, но на StockItems не имеет.
- Выбранная последовательность UPDLOCK+ блокировок:\
  1 - StockItems Id ASC\
  2 - Reservations Id ASC\
  3 - ReservationItems (ReservationId, ProductId) ASC (не соблюдается)
- Order блокировок записей в таблице не форсируется жестко, но запросы и индексы написаны так, что MS SQL обычно должен блокировать в нужной последовательности.
- Местами оптимистичная обработка конкуренции, местами пессимистичная.
- Не реализованы try и повторы запросов к СУБД.
- Текущую схему БД можно увидеть в файле `текущая схема базы данных.sql`
- Есть миграции, есть `docker-compose.yaml`. Применять миграции нужно руками (или с EF Core UI). Тесты сами развертывают СУБД в докере и применяют миграции (требуют настройки в `SqlServerDatabaseFixture.cs`), но запускаются только с хоста, для CI/CD не настроены.
- Юзкейсы с запросами к БД в `InventoryService.Api/Application/`, DbContext в `InventoryService.Api/Infrastructure/`, модели в `InventoryService.Api/Domain/`.
- Конкурентные запросы не протестированы.
